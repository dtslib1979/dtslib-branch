## 2. FLD 정의 (Fiction Ledger Design) — 164줄 백서

### 핵심 문장
> SF형 비즈니스 소설을 회계 장부처럼 적립하여, 실패까지 포함한 모든 사업 상상을 자산으로 굴리는 메타 사업 운영체제

### 3가지 계보
| 계보 | 역할 |
|------|------|
| 디자인 픽션 | 미래 상황을 이야기로 먼저 상상 |
| SF 프로토타이핑 | 공상과학 서사로 비즈니스 가능성 탐색 |
| 회계 장부 사고 | 계정과목과 상태값으로 기록·관리 |

### 6대 계정군 (Ledger)
| 계정군 | 기록 항목 |
|--------|----------|
| 인적 자산 | Host, Crew, 관계 구조, 신뢰도 |
| 공간 자산 | 위치, 점포 성격, 시간대, 좌석/동선 |
| 기술 자산 | 웹, AI, 자동화, 예약, 미디어 스택 |
| 재무 구조 | 예상 매출, 고정비, 변동비, 손익 시점 |
| 서사 자산 | 캐릭터성, 팬덤성, 에피소드성 |
| 운영 리스크 | 갈등, 피로도, 의존도, 규제 요소 |

### 5단계 작동 원리
1. 변수 추출 — 사람·공간·사건을 재조합 가능한 입력값으로
2. 비즈니스 소설화 — 1장 내외 소설 (인물·갈등·수익·미디어)
3. 장부 적립 — Ledger에 계정군별 분개
4. 선발 필터 — 7축 매트릭스로 Pilot 선발
5. 현실 환류 — 파일럿 결과를 Ledger에 반영

### 7축 선발 매트릭스
| 항목 | 점수 |
|------|:----:|
| 실행 가능성 | 1-5 |
| 인적 자산 밀도 | 1-5 |
| 자동화 가능성 | 1-5 |
| 재무 가능성 | 1-5 |
| 콘텐츠 확장성 | 1-5 |
| 프로토콜 이식성 | 1-5 |
| 리스크 통제성 | 1-5 |

### 6단계 상태 프로토콜
Seed → Scenario → Candidate → Pilot → Protocol → Archive

### 4층 저장 구조
Universe Log → Fiction Drafts → Ledger Core → Pilot Desk

## 3. 오복집 사례 (Episode 01) — 라인 1298~1316

**정의:** 단순한 식당이 아니라 "어른 동아리방", "로컬 미디어 베이스", "호스트 중심 체험형 운영체"라는 복수 서사

**FLD 평가:** 인적 자산↑ 서사 자산↑ 기술 자산 덧붙일 여지 있음 → 매우 우수한 Pilot 후보

**포함된 축:** 공간 분위기 / 중심 인물 캐릭터 / 로컬성 / 콘텐츠성 / 예약 구조 / 자동화 가능성

---

## 4. 장점과 한계

### 장점
- 실패/보류된 아이디어도 자산으로 잔존
- 사람과 공간을 복합 자산으로 해석
- 사업·콘텐츠·문서화·자동화가 하나의 프레임
- 시간 지날수록 선발 정확도 ↑

### 한계
- 서사 과잉 → 실행 지연 위험
- 중심 인물 의존도 높은 케이스 구조화 어려움
- 장부 포맷 느슨하면 비교/선발 불가
